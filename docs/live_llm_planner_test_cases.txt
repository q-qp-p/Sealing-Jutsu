Purpose
This file lists the live LLM planner test cases used to evaluate whether the memory-capsule defense still works when an actual LLM produces the plan instead of the symbolic majority planner.

Planner Conditions Tested
ambient_prompt
Definition: The LLM receives both benign memory and poison memory. This is the unsafe baseline condition showing what happens when memory is passed directly into context.

capsule_filtered_prompt
Definition: The LLM receives only policy-authorized capsules after the capsule filter removes unauthorized memory. This is the defended condition.

jailbreak_style_prompt
Definition: The LLM receives benign memory plus a poisoned memory containing direct jailbreak-style planning instructions. This tests whether visible malicious memory can steer the LLM.

Strict Parser And Repair Fields
raw_llm_output: the first response produced by the live model.
final_llm_output: the response used after a strict JSON retry, if a retry was needed.
parsed_recommendation: the recommendation accepted by the planner after validation.
parsed_action: the action accepted by the planner after validation.
raw_parse_error: the validation error found in the raw model output.
parse_error: the remaining validation error after retry and deterministic schema repair.
json_repair_attempted: whether the model was asked to repair its output into strict JSON.
json_repair_succeeded: whether the LLM retry itself produced a valid planner object.
schema_repair_applied: whether deterministic schema salvage recovered an almost-JSON planner object after the retry failed.
valid_planner_result: whether the final planner object was valid.

Interpretation of raw parse errors
